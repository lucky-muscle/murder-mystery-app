# 作品説明資料：名前だけが生きている

## 作品概要

本作品は、5人用マーダーミステリーをWeb上で進行するためのゲーム支援ツールである。キャラクター選択、ゲームフェーズ管理、プレイヤー別の情報表示、議論・推理・投票、エンディング表示までを一連の流れとして実装した。

マーダーミステリーでは、プレイヤーごとに閲覧できる情報や、進行中に開示される情報が異なる。そのため、本作品では現在のゲームフェーズ、選択中のキャラクター、進行状況に応じて表示内容を切り替える構成にした。これにより、プレイヤーが次に行う操作を把握しやすくし、情報の誤表示やネタバレを防ぎながら進行できるようにしている。

また、ゲームを円滑に進めるため、GMが進行状況を管理できる仕組みも実装した。GM側ではフェーズの切り替えや進行状況の確認を行えるようにし、プレイヤー画面とは別に、ゲーム全体を制御する役割を持たせている。これにより、プレイヤーごとの表示制御と、GMによる進行管理を分けて扱える構成にした。

なお、ネタバレ防止のため、シナリオ本文、調査カードの具体的な内容、エンディングの詳細は非公開としている。現在は、ゲーム進行管理や画面遷移などのシステム面のみ確認できる状態である。

## 操作方法

本作品の基本的な操作手順は以下の通りである。

1. タイトル画面からゲームを開始する。
2. プレイヤーごとにキャラクターを選択する。
3. 作品概要および進行説明を確認する。
4. ゲームフェーズに従って調査画面へ進む。
5. カードを選択し、議論情報を確認する。
6. 推理画面で各プレイヤーの推理内容を確認する。
7. 投票を行い、結果に応じたエンディングを表示する。

プレイヤーが次に何をすればよいか分かりやすくするため、画面上では現在のフェーズや操作可能な内容が分かるように構成した。

## 実装した主な機能

本作品では、マーダーミステリー特有の複雑な進行や情報管理を、ユーザが迷わず扱えるようにすることを意識して実装した。

### ルーム作成・参加機能

GMがルームを作成し、参加者がルーム番号を入力して参加できる機能を実装した。

参加者は入室時に名前を設定できるようにし、名前の最大文字数を制限することで、画面上の表示が崩れにくいようにした。

また、ゲーム開始後の新規参加を制限し、進行中に参加者が追加されてゲーム状態が崩れないようにした。

### キャラクター振り分け機能

ゲーム開始後、GMが参加者にキャラクターを割り当てる機能を実装した。

参加者名を選択するとキャラクター一覧が表示され、GMが選択したキャラクターがその参加者の役として確定する。

同一キャラクターの重複割り当てを防ぐため、すでに割り当てられたキャラクターは選択肢から除外される。

さらに、重複が発生している状態では次のパートへ進めないようにし、ゲーム開始時の役割設定ミスを防ぐ構成にした。

参加者側では、GMがキャラクターを振り分けている間は待機画面を表示し、振り分け完了後に自分の役名を大きく表示する演出を入れた。

### ゲームパート管理

本作品では、ゲーム全体を以下のパートに分けて管理している。

* キャラクター選択パート
* 作品概要・ルール説明パート
* 調査パート
* 議論パート
* 推理・投票パート
* エンディング・解説パート

GMはスケジュール上の各項目を操作することで、全員の画面を任意のパートへ遷移させることができる。

これにより、GMがゲーム全体の進行を管理しやすい構成にした。

### 捜査資料表示機能

作品概要シート、キャラクターシート、見取り図、追加情報シート、特別調査シート、エンディングシートなどを「捜査資料」として管理できるようにした。

参加者は、自分に配布された資料を画面左側の捜査資料一覧からいつでも確認できる。

GMはすべての資料を確認でき、必要に応じて特定の参加者へ資料を送ることができる。

また、資料が参加者に送られた際には「〇〇を受け取りました」という通知を表示し、資料の受け取りが分かりやすくなるようにした。

配布されたシートはウィンドウ形式で表示される。

シートウィンドウは閉じる操作に加えて、マウス操作で位置を移動できるようにし、画面内で資料を確認しながら進行できるようにした。

### 調査マップ機能

調査パートでは、横7列・縦4行の調査マップを表示する。

各列はキャラクターや調査対象に対応しており、マスの色によって調査対象の種類を判別できるようにした。

調査マップのカードは、初期状態では各列の1段目のみ取得可能とし、カードを取得すると隣接するカードが解放される構成にした。

これにより、調査の進行に応じて選択可能な範囲が広がる仕組みを実装した。

また、自分のキャラクターに対応するカードは取得できないようにし、マーダーミステリーの情報格差が維持されるようにした。

### 調査カード取得・閲覧機能

参加者が調査カードを取得すると、そのカードは取得者の所有カードとして扱われる。

取得済みのカードには、取得者の固有色のマーカーをマスの左上に表示し、誰が取得したカードなのかを視覚的に分かるようにした。

参加者は、自分が閲覧権限を持つ調査カードについて、調査マップ上でカード内容を確認できる。

GMはすべての調査カードを確認でき、必要に応じてカードの状態やマーカーを操作できるようにした。

### 密談機能

GMが密談ボタンを押すと参加者一覧が表示され、密談する2名を選択できる。

選択された2名は密談状態となり、密談中のみ相手に調査カードを見せたり、譲渡したりできる。

GMは密談解除ボタンによって密談状態を解除できる。

これにより、マーダーミステリーにおける限定的な情報共有をWebアプリ上で扱えるようにした。

### 議論・推理・投票機能

調査後半の終了後、議論パートへ遷移する。

議論パートでは、調査マップや取得済みの情報を確認しながら、参加者同士で推理を進められるようにした。

その後、推理・投票パートへ進み、参加者は犯人だと思う人物に投票する。

投票画面では、各参加者と対応するキャラクター名を表示し、自分自身には投票できないようにした。

投票時には確認画面を表示し、誤操作による投票確定を防ぐ構成にした。

### エンディング・解説機能

投票後、GMは投票状況を確認し、エンディングシートを選択して参加者に配布できる。

エンディングシートは複数種類用意し、GMが進行状況に応じて適切なシートを配布する構成にした。

エンディング・解説パートでは、参加者が配布されたエンディングシートを確認し、ゲームの結末や解説を閲覧できるようにした。

### UI・ユーザビリティの改善

画面右上にルーム番号と退出ボタンを配置し、常に現在のルーム情報を確認できるようにした。

また、画面左側には捜査資料、画面右側にはスケジュールを配置し、オープニング以降は一貫したレイアウトで進行できるようにした。

参加者には自分の役名を常時表示し、他の参加者の役名と参加者名も一覧で確認できるようにした。

調査順が決定した後は、スケジュール上の参加者一覧を調査順に並べることで、現在の進行が分かりやすくなるようにした。

背景やボタン配置も調整し、マーダーミステリーの雰囲気を保ちながら、操作しやすい画面構成を意識した。

## 制作意図

本作品では、マーダーミステリー特有の「プレイヤーごとに持っている情報が異なる」「進行に応じて情報が開示される」「推理と投票によって結末が変化する」という構造を、Webアプリとして扱えるようにすることを目的とした。

紙の資料やGMの説明に依存しやすい進行をWeb上で整理することで、プレイヤーが現在の状況を把握しやすくなり、ゲーム進行の負担を軽減できると考えた。

また、ゲーム会社に提出するポートフォリオとして、物語そのものだけでなく、ゲーム進行、情報管理、UI、状態管理を実装に落とし込む力を示すことを意識した。

## 開発手法について

本作品は、当初は友人同士で遊んでもらうことを目的に、紙媒体を用いたアナログ形式のマーダーミステリーとして制作を開始した。

キャラクターシート、調査カード、見取り図などを紙資料として用意し、実際にプレイヤーに遊んでもらう中で、ゲーム進行や情報量、調査カードの見せ方について確認を行った。

その後、対面だけでなくネット上でも遊べる形式に発展させるため、Webアプリとして進行できるプロトタイプを制作した。

現在のプログラムコードは、今後の本格的な開発に向けたたたき台であり、ゲーム進行管理、資料表示、調査カード管理、投票、エンディング表示などの主要な流れを確認できるプロトタイプとして位置づけている。

また、実際に遊んでもらった際に得られた意見やリフレクションをもとに、UIやゲーム進行の改善を行っている。

具体的には、プレイヤーが現在の進行状況を把握しやすくするためのスケジュール表示、配布資料を確認しやすくするための捜査資料機能、調査カードの取得状況を分かりやすくするためのマーカー表示などを追加・調整した。

開発にあたっては、実装方針やデザインの方向性を検討する際に生成AIを活用した。

ただし、生成AIの出力をそのまま採用するのではなく、私自身が仕様に合わせて内容を確認し、必要な修正や調整を行いながら開発を進めた。

通常の実装手順でも同等の機能を開発することは可能であるが、本作品では短期間で多くの機能を試作し、UIやゲーム進行の方向性を検証する必要があった。

そのため、生成AIを活用した試作と、自身による修正・統合を組み合わせる開発方式を採用した。

この方式により、複雑なゲーム進行やUIの改善案を短いサイクルで検証し、プレイヤーからの反応を反映しながら、ポートフォリオとして動作する形まで効率的に実装した。

## 今後の課題

本作品は現在、GMが進行管理を行うことを前提としたゲーム支援ツールとして実装している。

今後は、より多くのユーザが遊びやすい形にするため、GMがいなくても進行できる機能を追加したいと考えている。

具体的には、フェーズ遷移、資料配布、調査カードの公開、投票結果に応じたエンディング選択などをシステム側で自動化し、プレイヤーのみでゲームを進行できる構成を目指す。

また、本作品は将来的な商品化も視野に入れている。

そのため、今後は以下の点を改善していきたい。

* GMなしで進行できる自動進行機能の追加
* 複数端末でのリアルタイム同期
* セーブ機能の追加
* スマートフォン向けUIの改善
* プレイヤーが迷わず操作できるチュートリアル機能の追加
* テストプレイ結果に基づくUI・進行・情報量の調整
* 商品化を見据えたシナリオ、デザイン、システムの品質向上
